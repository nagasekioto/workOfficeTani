💾 バックアップ手順（1-7-5）

← その他に戻る
このシステムは最終的に別のパソコン（本番機）で稼働させる想定です。このページの①は、 本番機で最初にセットアップする際に、ゼロから同じ手順を行うための作業手順です。 ②以降は、セットアップ後の日常的な操作（手動バックアップ・復元・タスク管理）です。
✅ 動作確認済みの実績（2026年7月9日・開発機にて）
下記①の手順で毎日23:30の自動バックアップを設定し、復元テストも実施済み （求職者12件・氏名・電話番号を含め正しく復元できることを確認）。 その後、毎月15日 23:30に実行する方式に変更（保存先も C:\workOfficeTani\backup に変更）。
① 初期セットアップ手順（本番機で最初に1回行う）
1PowerShellを「管理者として実行」で開く
⚠️ この後の手順6（タスク登録）で管理者権限が必要になるため、最初から管理者権限のPowerShellを 使うことで、途中で画面を開き直す手間を省けます。
  1. スタートメニューを開き、「PowerShell」と検索する
  2. 表示された「Windows PowerShell」を右クリック→「管理者として実行」をクリック
  3. 「このアプリがデバイスに変更を加えることを許可しますか？」と出たら「はい」をクリック
以降の手順②〜⑨は、すべてこの管理者権限のPowerShell画面でそのまま行ってください。
2バックアップ保存フォルダを作る
New-Item -ItemType Directory -Path "C:\workOfficeTani\backup" -Force
保存先は C:\workOfficeTani\backup（workOfficeTani配下の backup フォルダ）です。 フォルダが存在しない場合は上記コマンドで新規作成されます。既に存在する場合は -Force により何も壊さずそのまま使われます（バックアップスクリプト自体も 実行のたびに同様のフォルダ作成チェックを行うため、途中でフォルダを消してしまっても 次回実行時に自動的に作り直されます）。
3PostgreSQLの実行ファイルの場所を確認する
開発機では C:\workofficetani\bin にありましたが、本番機ではインストール場所が 異なる可能性があります。まず標準的な場所を確認してください。 
Get-ChildItem "C:\Program Files\PostgreSQL" -Directory -ErrorAction SilentlyContinue
見つからない場合は、パソコン全体から検索します（少し時間がかかります）。 
Get-ChildItem -Path C:\ -Filter "pg_dump.exe" -Recurse -ErrorAction SilentlyContinue -Force
表示された pg_dump.exe があるフォルダのパスを、次の手順4で使う $pgBin の値として使ってください。
4バックアップ用スクリプトを作る
⚠️ 重要：この後の内容をメモ帳等に貼り付けて保存すると、引用符が自動的に変換されて 「文字列に終端記号 " がありません」というエラーになることがあります。 必ずPowerShell上でこのコマンドをそのまま実行して、ファイルを直接作成してください （日本語コメントを含めないのも、文字化けを避けるポイントです）。 
Remove-Item C:\workOfficeTani\backup\backup.ps1 -ErrorAction SilentlyContinue
Add-Content -Path C:\workOfficeTani\backup\backup.ps1 -Value '$backupDir = "C:\workOfficeTani\backup"'
Add-Content -Path C:\workOfficeTani\backup\backup.ps1 -Value 'New-Item -ItemType Directory -Path $backupDir -Force | Out-Null'
Add-Content -Path C:\workOfficeTani\backup\backup.ps1 -Value '$pgBin = "（手順3で確認したフォルダのパス）"'
Add-Content -Path C:\workOfficeTani\backup\backup.ps1 -Value '$dbName = "kaseihu"'
Add-Content -Path C:\workOfficeTani\backup\backup.ps1 -Value '$dbUser = "postgres"'
Add-Content -Path C:\workOfficeTani\backup\backup.ps1 -Value '$env:PGPASSWORD = "7136"'
Add-Content -Path C:\workOfficeTani\backup\backup.ps1 -Value '$dateStr = Get-Date -Format "yyyyMMdd_HHmmss"'
Add-Content -Path C:\workOfficeTani\backup\backup.ps1 -Value '$backupFile = "$backupDir\kaseihu_$dateStr.backup"'
Add-Content -Path C:\workOfficeTani\backup\backup.ps1 -Value '& "$pgBin\pg_dump.exe" -U $dbUser -h localhost -d $dbName -F c -f $backupFile'
Add-Content -Path C:\workOfficeTani\backup\backup.ps1 -Value 'if ($LASTEXITCODE -eq 0) { Write-Output "Backup OK: $backupFile" } else { Write-Output "Backup FAILED, exit code: $LASTEXITCODE" }'
Add-Content -Path C:\workOfficeTani\backup\backup.ps1 -Value 'Get-ChildItem -Path $backupDir -Filter "kaseihu_*.backup" | Sort-Object LastWriteTime -Descending | Select-Object -Skip 12 | Remove-Item -Force'
$pgBinの行だけ、実際に確認したパスに書き換えてから実行してください。 実行後、中身が文字化けせず正しく書き込まれたか確認します。 
Get-Content C:\workOfficeTani\backup\backup.ps1
最後の1行は保持ロジックです。毎月15日に1回だけ実行する運用のため、日数ではなく 「直近12個（約1年分）を残し、それより古いものだけ削除」という件数ベースの方式に しています（日数ベースの「30日より古いものを削除」のままだと、次回実行時に前回分が 必ず削除対象になってしまい、バックアップが1世代しか残らなくなるためです）。
5実行ポリシーを設定する
初期状態のWindowsではスクリプトの実行が禁止されているため、まず許可します。 
Set-ExecutionPolicy -ExecutionPolicy RemoteSigned -Scope CurrentUser -Force
6手動で1回試す
& "C:\workOfficeTani\backup\backup.ps1"
「Backup OK: （ファイルパス）」と表示され、C:\workOfficeTani\backup フォルダにファイルができていれば成功です。
7毎月15日に自動実行するタスクを登録する
PowerShellのNew-ScheduledTaskTriggerには「毎月◯日」を直接指定する機能が ないため、ここだけはschtasksコマンドを使います。 
schtasks /create /tn "kaseihu-DB-Backup" /tr "powershell.exe -ExecutionPolicy Bypass -File \"C:\workOfficeTani\backup\backup.ps1\"" /sc MONTHLY /d 15 /st 23:30 /rl HIGHEST /f
実行時刻（23:30の部分）・実行日（15の部分）は、業務に支障のない値に変更しても構いません。
手順1で管理者権限のPowerShellを開いていない場合、ここで「アクセスが拒否されました」という エラーになります。その場合は手順1からやり直してください。
8タスクが正しく動くか確認する
時刻を待たずに、今すぐ手動でタスクを起動して確認できます。 
Start-ScheduledTask -TaskName "kaseihu-DB-Backup"
Start-Sleep -Seconds 5
Get-ChildItem C:\workOfficeTani\backup
新しいタイムスタンプのバックアップファイルが増えていれば、タスク経由でも正常に動作しています。
9復元テストを行う（必須）
「バックアップは取れていたのに、いざという時に復元できなかった」という事態を防ぐため、 必ず一度、実際のデータに触れない方法で復元テストを行ってください。手順は下記④を参照してください。
② 今すぐ手動でバックアップを取りたいとき
PowerShellを開いて、以下を実行してください。 
& "C:\workOfficeTani\backup\backup.ps1"
「Backup OK: （ファイルパス）」と表示されれば成功です。
③ データを復元したいとき（本番復元）
⚠️ この操作は既存のデータを上書きします。本当に必要なとき以外は実行しないでください。 心配な場合は、先に④の「試験復元」で内容を確認してから行ってください。
  1. C:\workOfficeTani\backup フォルダの中から、復元したいバックアップファイル（例：kaseihu_20260709_215908.backup）を確認する
  2. PowerShellで以下を実行する（ファイル名・pgBinのパスは実際のものに置き換える）
$env:PGPASSWORD = "7136"
& "C:\workofficetani\bin\pg_restore.exe" -U postgres -h localhost -d kaseihu --clean "C:\workOfficeTani\backup\kaseihu_20260709_215908.backup"
--clean を付けると、既存のテーブルを消してからバックアップ内容で復元し直してくれます。
④ 試験復元（本番データに触れずに確認したいとき）
「本当にこのバックアップから復元できるか」を、実際の kaseihu データベースに触れずに確認する方法です。
1. 試験用の空のデータベースを作る
$env:PGPASSWORD = "7136"
& "C:\workofficetani\bin\createdb.exe" -U postgres -h localhost kaseihu_test
2. バックアップファイルを試験用DBに復元する
& "C:\workofficetani\bin\pg_restore.exe" -U postgres -h localhost -d kaseihu_test "C:\workOfficeTani\backup\（確認したいファイル名）"
3. 中身を確認する（件数比較・データ内容の目視確認）
& "C:\workofficetani\bin\psql.exe" -U postgres -h localhost -d kaseihu -c "SELECT count(*) FROM persons;"
& "C:\workofficetani\bin\psql.exe" -U postgres -h localhost -d kaseihu_test -c "SELECT count(*) FROM persons;"
& "C:\workofficetani\bin\psql.exe" -U postgres -h localhost -d kaseihu_test -c "SELECT last_name_kanji, first_name_kanji, mobile_phone FROM persons LIMIT 3;"
件数が一致し、氏名・電話番号などの中身も正しく表示されればOKです。
4. 試験用データベースを削除する（後片付け）
& "C:\workofficetani\bin\dropdb.exe" -U postgres -h localhost kaseihu_test
⑤ 設定内容の一覧（開発機での実績値）
項目内容
バックアップ対象データベース kaseihu（求職者・求人者・売上・紹介状など全データ）
バックアップスクリプトC:\workOfficeTani\backup\backup.ps1
実行タイミング毎月15日 23:30（タスクスケジューラ「kaseihu-DB-Backup」）
保存先フォルダC:\workOfficeTani\backup（存在しない場合はスクリプト実行時に自動作成）
ファイル名の形式kaseihu_（年月日_時分秒）.backup
保持期間直近12個（約1年分）。それより古いものは自動削除
PostgreSQLツールの場所（開発機の例）C:\workofficetani\bin（本番機では①手順3で確認し直すこと）
⚠️ バックアップ先を同じパソコンの中だけにしていると、パソコンの故障・盗難・火災等が起きた場合は 元データもバックアップも同時に失われます。①手順1で触れた通り、クラウド同期フォルダの中に 保存先を作ることを検討してください。
⑥ タスクの確認・変更・削除
やりたいことコマンド
タスクの状態を確認Get-ScheduledTask -TaskName "kaseihu-DB-Backup"
今すぐ手動で実行Start-ScheduledTask -TaskName "kaseihu-DB-Backup"
タスクを削除Unregister-ScheduledTask -TaskName "kaseihu-DB-Backup" -Confirm:$false
実行日・実行時刻を変更したい場合は、①手順7の登録コマンドの /d 15（実行日）や /st "23:30"（実行時刻）の部分を変えて、 同じコマンドをもう一度実行してください（/fにより上書き登録されるため、 事前にタスクを削除する必要はありません。管理者権限のPowerShellが必要です）。
